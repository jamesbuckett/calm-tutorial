Summary

TL;DR — Summary

  • One artefact. The architecture and its controls live in one architecture.calm.json, validated against its pattern and the shared meta-schema.
  • PR-time enforcement. calm validate runs next to your unit tests; non-conforming changes fail the build, not the next Architecture Review Board.
  • Regulator-ready. Controls map to DORA (EU), PCI DSS v4.0.1, MAS TRM (SG), APRA CPS 234 (AU), HKMA TM-G-1 (HK), and OSFI B-13 (CA).
  • Tooling exists today. Install @finos/calm-cli from npm, render with a self-hosted CALM Hub, ship a docs site with calm docify.

Framing sentence from the source: CALM turns architecture into a versioned JSON document that travels with the code, validates in CI, and binds compliance controls to the relationships they govern.

01 · Introduction

Architecture lives where code lives.

The status quo: drift by design

  • Approved by a board, drawn in Visio, attached to an ADR — and stale within two sprints
  • Reviews happen in meetings that produce no machine-checkable output
  • Compliance evidence is forensic — screenshots, change tickets, runbooks for DORA Art. 6, SOX ITGC, MAS TRM

The CALM answer: one document, one truth

  • architecture.calm.json binds intent (pattern), instantiation (deployed system), and control (compliance posture)
  • Review it in a pull request, validate in CI, render as C4 diagrams, query for drift
  • One artefact answers: what is the system? does it meet our golden pattern? which controls apply?

For three decades, architecture artefacts have lived in Visio files, Confluence pages, and PowerPoint decks while implementation lived in Git — the two drift apart at every release. CALM closes the gap by treating the model itself as a versioned JSON document that travels with the source. Every Tier 1 institution has the same problem.

02 · Core Concepts

The vocabulary of the spec.

  • Seven primitives — each a typed JSON object with a unique identifier and a defined slot in the meta-schema
  • Node · Relationship · Metadata · Pattern · Architecture · Control · Flow — learn these and you can read almost any CALM document
  • Interfaces and evidence, defined in the glossary, complete the set
  • Flow — ordered sequence of interactions: a business or data path across nodes — register an attendee, settle a trade, process an instant payment — referencing existing nodes and relationships in order
  • A minimal model, visualised: three nodes, two relationships, one control — renderable as a C4 container diagram or a Mermaid graph; CALM is the source, the diagrams are views

CALM has a small, deliberate set of primitives. Flows stay consistent with the structural model because they only reference nodes and relationships that already exist. The next six slides take the primitives one at a time, with the field list from the source's concept cards.

02 · Core Concepts

Node — typed system entity

  • A service, database, network, actor, deployment target, or other addressable element
  • Identified by a stable unique-id, and exposing interfaces — typed connection points such as a URL or host/port pair

02 · Core Concepts

Pattern — reusable template

  • A JSON Schema that describes a class of valid architectures — a golden three-tier web app, an event-driven microservice topology
  • Patterns are the unit of reuse

A pattern captures what is non-negotiable about a class of systems. Teams instantiate it; the pattern enforces required node and relationship types and the controls that must be present.

02 · Core Concepts

Architecture — concrete instantiation

  • The same nodes and relationships, populated with real names, owners, and runtime details

Pattern is the class, architecture is the instance. The architecture document is the thing you review in a PR and diff against the live estate.

02 · Core Concepts

Control — compliance / security check

  • Declarative requirements attached to nodes or relationships
  • “All relationships entering PCI scope must use mutual TLS (mTLS)” · “every persistent store must declare encryption at rest”
control-id
string
requirement
object

Controls are what make CALM interesting to regulated institutions: the requirement lives on the exact node or relationship it governs, and evidence links back from the same place.

03 · Why It Matters

For Tier 1 financial institutions, this is overdue.

  • Controls become first-class. DORA Art. 6, PCI DSS v4.0.1 Req. 4, SWIFT CSCF 2.4A attach to the connections they constrain — evidence is a query, not a reconstruction
  • Reality vs. intent, diffed. A nightly job renders the live topology as CALM; diff against the approved document → drift PR opened automatically
  • Architecture review = PR review. The JSON diff, the rendered C4 view, and the controls delta — in the same place reviewers see code
  • Golden patterns enforced at PR time. A new service must validate against the approved pattern before merge; production review arrives pre-validated

FINOS members did not open-source CALM to make documentation prettier. They open-sourced it because regulators are asking the same questions in five jurisdictions, and there is no scalable way to answer them with PowerPoint. Drift detection introspects the cluster, Terraform state, or cloud APIs.

03 · Why It Matters

The CI/CD checkpoint

  • calm validate runs on every pull request and blocks the merge on failure — the same gate that catches a failing unit test or SAST finding
  • Controls map to the evidence requirements of DORA (EU), MAS TRM (Singapore), APRA CPS 234 (Australia), HKMA SPM TM-G-1 (Hong Kong), and OSFI B-13 (Canada)
  • The model is the audit trail
  • CALM ships the controls mechanism — requirement and configuration schemas, documented at calm.finos.org; the regulatory mappings are yours to author

The point to land: validation is not a separate governance process — it is the same merge gate the rest of the codebase already uses. The five-jurisdiction mapping is the business case for Tier 1 institutions.

04 · Installation

Install on Ubuntu 26.04 LTS (1/4)

Step 01 — preflight. The only hard prerequisite is Node.js 20+; Java 21 and Maven matter only if you build CALM Hub from source. Skip any step whose binary already reports ok.

# report which prerequisites are present on this machine
for cmd in curl gpg make git node npm calm java mvn; do
  if command -v "$cmd" >/dev/null 2>&1; then
    ver=$("$cmd" --version 2>/dev/null | head -n1)
    printf "  ok      %-6s %s\n" "$cmd" "$ver"
  else
    printf "  missing %-6s -\n" "$cmd"
  fi
done

bash · a fresh Ubuntu 26.04 box typically reports curl, gpg, git ok — make, node, npm, calm, java, mvn missing

Run the loop once; it probes every binary the rest of the section installs. Re-check versions, not just presence: Node must be 20+ and Java 21+ for CALM Hub — an older binary on PATH counts as missing. Step 02 covers curl, gpg, make, git; step 03 node and npm; step 05 calm; step 07 (optional) java and mvn.

04 · Installation

Install on Ubuntu 26.04 LTS (2/4)

  1. Update apt and install prerequisites — sudo apt update && sudo apt install -y curl ca-certificates gnupg build-essential git
  2. Install Node.js 22 from NodeSource — apt's Node tracks Debian's snapshot and can drift; pin the LTS line: curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash - then sudo apt install -y nodejs
    • The documented floor is Node 20+ — Node 20 reached end-of-life in April 2026; Node 22 is maintenance LTS until April 2027
  3. Verify Node and npm — node --version → v22.14.0 · npm --version → 10.9.2

ca-certificates is preinstalled on Ubuntu Server images, so step 02 is skippable when preflight reported ok for curl, gpg, make, and git. Node 24 is the current active LTS if you prefer it; 22 is the conservative pin.

04 · Installation

Install on Ubuntu 26.04 LTS (3/4)

  1. Install the CALM CLI globally — npm install -g @finos/calm-cli; the binary is exposed as calm
    • EACCES? Never sudo npm — install Node via nvm, or set an npm user prefix and add it to PATH
  2. Verify the CLI — calm --version · calm --help lists the commands: generate · validate · diff · template · docify · init-ai

The user-prefix recipe: npm config set prefix ~/.npm-global, then add $HOME/.npm-global/bin to PATH. The six sub-commands map to the rest of the deck — generate and validate drive the tutorial, docify renders the docs site, diff powers drift detection, init-ai drops AI-assistant configuration into a repo.

04 · Installation

Install on Ubuntu 26.04 LTS (4/4)

  1. Optional — Java 21 and Maven for CALM Hub — needed only to build the Hub from source (a Quarkus app: Java 21+, Maven 3.8.6+): sudo apt install -y openjdk-21-jdk maven
    • The published Docker image needs neither — run it instead (see the tutorial)
  2. Optional — clone the reference repo — git clone https://github.com/finos/architecture-as-code.git; calm/getting-started carries conference-signup.pattern.json

The FINOS monorepo carries the spec, reference patterns, and the conference-signup tutorial used in the next section. Verify with java --version and mvn --version after installing.

04 · Installation

Common installation issues

  • EACCES on npm install -g — use nvm or an npm user prefix; never sudo npm, which writes into dpkg-managed paths npm cannot cleanly track
  • calm: command not found — run npm config get prefix and confirm <prefix>/bin is on PATH; with a user prefix, export it in ~/.bashrc
  • Node version mismatch — the documented floor is Node 20+; replace an older apt Node with NodeSource, or nvm install 22 && nvm use 22
  • Behind a corporate proxy — set HTTPS_PROXY / HTTP_PROXY / NO_PROXY, plus npm config set proxy and https-proxy

The user-prefix PATH line: export PATH="$HOME/.npm-global/bin:$PATH" in ~/.bashrc. Proxy settings need to exist in the shell before npm install runs, and npm keeps its own proxy config besides the environment variables.

05 · Hands-On Tutorial

Conference signup, end to end (1/2)

Locate → generate → validate → visualise → docify — the same loop you will run in CI.

  1. Locate the pattern — the monorepo's calm/getting-started bundles conference-signup.pattern.json: a JSON Schema describing the shape of a valid architecture
  2. Generate a concrete architecture — calm generate --pattern conference-signup.pattern.json --output conference-signup.arch.json → a stub with five nodes, four relationships, and placeholders ([[ HOST ]], -1)
  3. Validate — calm validate --pattern … --architecture …; the exit code is non-zero only on errors, --strict fails warnings too — that combination is the CI gate

Warnings on a fresh stub are expected — spectral reports architecture-has-no-placeholder-properties-string for each placeholder you have not filled in yet (hasErrors: false, hasWarnings: true). Generate logs: loading the pattern as file, then "Successfully generated architecture".

05 · Hands-On Tutorial

Conference signup, end to end (2/2)

  1. Visualise with CALM Hub — the self-hostable registry and visualisation server: upload the architecture and get a rendered topology and a shareable URL. Quickest is the published Docker image: cd calm-hub/deploy && docker compose up; from source: ./mvnw -pl calm-hub quarkus:dev -Dcalm.database.mode=standalone → http://localhost:8080
  2. Render a documentation site — calm docify --architecture conference-signup.arch.json --output ./website — a static site of nodes, relationships, controls, and embedded diagrams, ready for GitHub Pages

The Docker route needs no Java at all; building from source needs Java 21 + Maven (install step 07). Standalone storage mode also removes the MongoDB dependency in dev. Startup line: calm-hub on JVM (powered by Quarkus 3.x), listening on localhost:8080.

What the architecture document looks like (1/2)

{
  "$schema": "https://calm.finos.org/getting-started/conference-signup.pattern.json",
  "nodes": [
    { "unique-id": "conference-website", "node-type": "webclient",
      "interfaces": [{ "unique-id": "conference-website-url", "url": "[[ URL ]]" }] },
    { "unique-id": "load-balancer", "node-type": "network",
      "interfaces": [{ "unique-id": "load-balancer-host-port",
                       "host": "[[ HOST ]]", "port": -1 }] },
    /* … "attendees" (service) and "attendees-store" (database) … */
    { "unique-id": "k8s-cluster", "node-type": "system", "name": "Kubernetes Cluster" }
  ]
}

conference-signup.arch.json — nodes, condensed · the $schema URL records which pattern the document instantiates

Note the interfaces on each node: typed connection points whose placeholder values ([[ HOST ]], -1) are exactly what the first-run validate warnings point at. The full document also carries metadata and controls; descriptions and two nodes are elided here.

What the architecture document looks like (2/2)

{
  "relationships": [
    { "unique-id": "conference-website-load-balancer", "protocol": "HTTPS",
      "relationship-type": { "connects": {
        "source":      { "node": "conference-website" },
        "destination": { "node": "load-balancer" } } } },
    /* … "load-balancer-attendees" (mTLS) and "attendees-attendees-store" (JDBC) … */
    { "unique-id": "deployed-in-k8s-cluster",
      "relationship-type": { "deployed-in": { "container": "k8s-cluster",
        "nodes": [ "load-balancer", "attendees", "attendees-store" ] } } }
  ]
}

conference-signup.arch.json — relationships · spec 1.0 shown; 1.2 current — verify paths at calm.finos.org

Two relationship-type shapes on display: connects (with protocol) and deployed-in (a container grouping three nodes). The source page pins spec release 1.0; releases 1.1 and 1.2 have shipped since, and exact $schema paths and field shapes may evolve between releases.

06 · Glossary

Terms, defined (2/5)

Relationship primitive
Typed edge between nodes: connects, deployed-in, composed-of, interacts, options
Metadata primitive
Free-form key/value extensions — cost centre, data classification, and similar
Pattern noun
Reusable JSON Schema describing a class of valid architectures; the unit of reuse
Architecture instance
A concrete document validating against a pattern — the deployable contract
Control primitive
Declarative compliance or security requirement — mTLS, encryption at rest, residency
Flow primitive
Ordered node interactions — trade settlement, payment authorisation, registration

Details condensed away: options marks pattern decision points. Metadata is the intended home for enterprise concerns. An architecture is populated with real ids, names, owners, and runtime detail. Controls examples include data residency constraints.

07 · Takeaways

Six things to remember.

  • Architecture is now code. Diagrams are derived views — if they disagree, the JSON wins
  • The meta-schema is the heart. Patterns, architectures, controls all validate against it
  • Patterns enable reuse. One golden pattern, dozens of architectures
  • Validate in CI/CD. calm validate sits next to your unit tests; non-conformance fails the build
  • Controls bake in compliance. DORA, PCI DSS, SOX, MAS TRM attach to the relationships they govern — evidence queryable, not forensic
  • Visualise with CALM Hub. Diagrams render from the model — no hand-drawn topology to maintain

Each one shifts how architecture is practised — not what is drawn. The schema is the only thing you need to learn deeply; the pattern enforces what is non-negotiable and the architecture fills in the rest; a non-conforming architecture fails the build, not the next architecture review board.
